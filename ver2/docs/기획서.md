# 대전 밴드 생태계 통합 플랫폼 기획서

> 대전 지역 밴드 · 동아리 · 공연장을 연결하는 조직 기반 공연·소통·예약 통합 플랫폼

---

## 1. 프로젝트 개요

### 1.1 배경 및 목적

대전 지역에는 대학 밴드 동아리, 인디 밴드, 소규모 공연장 등 활발한 밴드 생태계가 존재한다. 그러나 이들을 체계적으로 연결하는 시스템이 부재하여 모든 소통과 운영이 파편화된 채널(카카오톡, 인스타그램, 수기 관리)에 의존하고 있다.

본 플랫폼은 이러한 분산된 밴드 생태계를 **하나의 구조화된 시스템**으로 통합하여, 밴드 · 동아리 · 공연장 간의 원활한 연결과 운영 효율화를 목표로 한다.

### 1.2 핵심 가치

| 가치 | 설명 |
|------|------|
| **연결** | 밴드-동아리-공연장 간 구조적 연결 |
| **효율** | 공연 조율, 예약, 홍보 등 운영 프로세스 자동화 |
| **성장** | 밴드 활동 기록 축적 및 노출을 통한 생태계 성장 |

### 1.3 핵심 컨셉: 조직 계층 구조

본 플랫폼의 가장 중요한 개념은 **"밴드 위에 동아리/연합 단위가 존재한다"**는 것이다. 개별 밴드는 독립적으로도 존재할 수 있지만, 대부분의 밴드는 대학 동아리나 지역 연합 같은 **상위 조직**에 소속되어 활동한다. 이 계층 구조를 플랫폼의 핵심 데이터 모델로 삼아, 조직 단위의 관리·소통·일정 운영을 체계적으로 지원한다.

```
조직 (Organization)
 |
 +-- 산하 밴드 A
 |    +-- 공연 기록 1 -- 공연장 X 연결 -- 팬/방문자
 |    +-- 공연 기록 2 -- 공연장 Y 연결 -- 팬/방문자
 |
 +-- 산하 밴드 B
 |    +-- 공연 기록 3 -- 공연장 X 연결 -- 팬/방문자
 |
 +-- 조직 공지 게시판
 +-- MT / 일정 관리
 +-- 연결된 공연장 목록
```

이 구조에서 데이터는 **조직 -> 밴드 -> 공연 기록 -> 공연장 연결 -> 팬/방문자** 순서로 계층적으로 흐른다.

## 2. 문제 정의

### 2.1 현재 상황 (As-Is)

| 영역 | 현재 방식 | 문제점 |
|------|-----------|--------|
| 공연 조율 | 카카오톡 단체방 | 히스토리 유실, 참여자 관리 불가 |
| 동아리 MT/일정 | 개별 연락 | 일정 충돌, 참석 확인 번거로움 |
| 공연장 예약 | 전화/DM 수동 예약 | 가용 일정 확인 어려움, 이중 예약 위험 |
| 테크라이더 공유 | PDF 파일 전달 | 버전 관리 불가, 포맷 불일치 |
| 밴드 홍보 | 인스타그램 개별 운영 | 발견성 낮음, 통합 노출 불가 |
| 동아리 내 밴드 교류 | 구조 없음 | 산하 밴드 간 협업/교류 기회 부재 |

### 2.2 목표 상태 (To-Be)

- 공연 기획부터 예약, 홍보, 기록까지 **한 곳에서** 관리
- 동아리-밴드 간 **계층적 조직 구조** 지원
- 공연장 **실시간 예약 시스템** 운영
- 밴드 프로필 · 공연 이력 기반 **자동 노출 및 매칭**

## 3. 플랫폼 구조

### 3.1 조직 계층 구조

```
조직 (Organization)
├── 동아리 / 연합 단위
│   ├── 산하 밴드 A
│   │   ├── 멤버
│   │   ├── 공연 기록
│   │   └── 팬/팔로워
│   ├── 산하 밴드 B
│   └── ...
├── 공지 게시판
├── 일정 (MT, 정기 공연 등)
└── 연결된 공연장
```

**조직 유형 예시:**
- 충남대 ○○밴드 동아리
- 한밭대 밴드 연합
- 대전 인디 연합
- 지역 공연 기획팀

### 3.2 주요 엔티티 관계

```
[조직] 1:N [밴드]
[밴드] N:M [멤버(사용자)]
[밴드] 1:N [공연]
[공연] N:1 [공연장]
[공연장] 1:N [예약]
[사용자] -- 팬/팔로워 --> [밴드]
[사용자] -- 팬/팔로워 --> [조직]
[공연] -- 방문/관심 --> [사용자(팬/방문자)]
```

### 3.3 조직 단위 핵심 기능 상세

조직은 단순한 그룹이 아니라 **실질적인 운영 허브**로 기능한다.

| 기능 | 설명 |
|------|------|
| 산하 밴드 관리 | 밴드 등록/초대/탈퇴, 밴드별 활동 현황 대시보드 |
| 조직 공지 게시판 | 전체 공지, 산하 밴드 대상 공지, 댓글/읽음 확인 |
| MT/일정 관리 | MT, 합주, 정기공연 등 일정 등록, 참석 여부 투표, 캘린더 |
| 멤버 통합 관리 | 조직 내 전체 멤버 명단, 밴드별 소속 현황 |
| 공연장 연결 | 자주 이용하는 공연장과의 연결, 단체 예약 |

## 8. MVP 범위 (1차 개발)

### 8.1 MVP 핵심 기능

1. **회원가입/로그인** - 이메일 + 소셜 로그인
2. **밴드 프로필** - 기본 정보, 멤버, 장르 등록
3. **조직(동아리) 생성** - 산하 밴드 등록 및 공지 게시판
4. **공연 등록/조회** - 공연 정보 등록 및 피드 노출
5. **공연장 등록/예약** - 공연장 프로필 + 예약 요청/승인 플로우

### 8.2 MVP 제외 항목 (2차 이후)

- AI 리마스터
- 후원 기능
- 공연 통계 분석
- 티켓 판매
- 테크라이더 자동 대조

## 9. 화면별 기능 명세

### 9.1 홈 피드

**목적:** 사용자가 플랫폼에 진입했을 때 가장 먼저 보는 화면. 대전 밴드 생태계의 최신 소식을 한눈에 파악할 수 있도록 한다.

**UI 구성:**

| 영역 | 요소 | 설명 |
|------|------|------|
| 상단 바 | 로고, 검색 아이콘, 알림 아이콘, 프로필 아이콘 | 검색은 밴드/공연/공연장 통합 검색 |
| 카테고리 탭 | [전체] [공연] [밴드 소식] [공연장] | 탭 전환으로 피드 필터링 |
| 다가오는 공연 섹션 | 가로 스크롤 카드 리스트 | 포스터 썸네일, 공연명, 일시, 공연장명 표시. 탭하면 공연 상세로 이동 |
| 추천 밴드 섹션 | 가로 스크롤 원형 프로필 리스트 | 밴드 프로필 사진, 밴드명, 장르 태그. 탭하면 밴드 프로필로 이동 |
| 피드 타임라인 | 세로 스크롤 카드 리스트 | 밴드 활동 소식(공연 등록, 새 음원 등), 공연 후기, 조직 공지 등. 각 카드에 좋아요/댓글/공유 버튼 |
| 하단 네비게이션 | [홈] [탐색] [공연장] [마이] | 4개 탭 고정 하단바 |

**사용자 인터랙션:**
- 피드 무한 스크롤 (페이지네이션)
- 카드 탭 시 상세 페이지 이동
- 팔로우한 밴드의 소식 우선 노출
- 당겨서 새로고침 (Pull to Refresh)

### 9.2 밴드 프로필

**목적:** 밴드의 모든 정보를 한 페이지에서 확인. 밴드의 포트폴리오이자 공식 페이지 역할.

**UI 구성:**

| 영역 | 요소 | 설명 |
|------|------|------|
| 헤더 | 커버 이미지, 프로필 사진, 밴드명, 장르 태그 | 팔로우 버튼, 후원 버튼(프리미엄) 포함 |
| 소개 섹션 | 밴드 소개 텍스트, SNS 링크 아이콘들 | 인스타그램, 유튜브, 사운드클라우드 등 |
| 소속 조직 뱃지 | 조직명 + 링크 | 탭하면 조직 페이지로 이동 |
| 멤버 섹션 | 멤버 프로필 카드 리스트 | 이름, 파트(보컬/기타/베이스/드럼 등), 프로필 사진 |
| 탭 영역 | [공연 이력] [포트폴리오] [테크라이더] | 탭 전환 |
| 공연 이력 탭 | 타임라인 형태 공연 카드 리스트 | 날짜, 공연명, 공연장명, 포스터 썸네일. 최신순 정렬 |
| 포트폴리오 탭 | 그리드 형태 미디어 리스트 | 사진/영상 썸네일. 탭하면 전체 화면 재생/뷰 |
| 테크라이더 탭 | 테크라이더 정보 표시 | 필요 장비 목록, 세팅 요구사항 (밴드 관리자만 편집 가능) |

**사용자 인터랙션:**
- 팔로우/언팔로우 토글
- 밴드 관리자에게만 편집 버튼 노출
- 공연 이력 카드 탭 시 공연 상세로 이동
- 공유 버튼으로 외부 공유 (딥링크 또는 URL)

### 9.3 조직 페이지

**목적:** 동아리/연합의 관리 허브. 산하 밴드를 한눈에 보고 조직 차원의 소통과 일정을 관리한다.

**UI 구성:**

| 영역 | 요소 | 설명 |
|------|------|------|
| 헤더 | 조직 커버 이미지, 로고, 조직명, 유형 태그(동아리/연합) | 가입 신청 버튼 (비멤버용) |
| 조직 소개 | 소개 텍스트, 설립 연도, 소속 학교/지역, SNS 링크 | - |
| 산하 밴드 목록 | 밴드 카드 그리드 | 밴드 프로필 사진, 밴드명, 장르, 활동 상태(활동중/휴식). 탭하면 밴드 프로필로 이동 |
| 탭 영역 | [공지] [일정] [멤버] | 탭 전환 |
| 공지 탭 | 게시글 리스트 | 제목, 작성자, 날짜, 읽음 수. 탭하면 상세 보기. 관리자만 작성 가능 |
| 일정 탭 | 캘린더 뷰 + 리스트 뷰 전환 | MT, 정기공연, 합주 등 일정 표시. 참석 여부 투표 버튼 |
| 멤버 탭 | 멤버 리스트 | 이름, 역할(관리자/멤버), 소속 밴드. 관리자는 권한 변경/추방 가능 |

**사용자 인터랙션:**
- 조직 관리자: 밴드 추가/제거, 공지 작성, 일정 등록, 멤버 관리
- 일반 멤버: 공지 조회, 일정 참석 여부 응답, 댓글
- 비멤버: 조직 정보 조회, 가입 신청

### 9.4 공연 상세

**목적:** 개별 공연의 모든 정보를 제공. 공연 전에는 홍보 페이지, 공연 후에는 아카이브 역할.

| 출연진 섹션 | 밴드 프로필 카드 리스트 | 출연 순서, 밴드명, 장르. 탭하면 밴드 프로필로 이동 |
| 상세 설명 | 공연 소개 텍스트 | 마크다운 또는 리치텍스트 |

| 공연 후 섹션 | 사진/영상 갤러리, 후기 | 공연 완료 후에만 노출 |

**사용자 인터랙션:**
- 관심 표시(북마크) 토글
- 캘린더 앱에 일정 추가 (ical 형식 내보내기)
- 공유 (URL 복사, SNS 공유)
- 밴드 관리자: 공연 정보 편집, 공연 후 사진/영상 업로드

**목적:** 공연장 정보 확인 및 예약 진입점. 밴드가 공연장을 탐색하고 예약을 결정하는 데 필요한 모든 정보 제공.

## 11. DB 스키마 초안

PostgreSQL + Prisma 기준 핵심 테이블 정의이다.

```prisma
// ============================================
// 사용자
// ============================================
model User {
  id            String   @id @default(cuid())
  email         String   @unique
  password      String?  // 소셜 로그인 시 null
  name          String
  nickname      String?
  profileImage  String?
  bio           String?
  provider      String?  // kakao, google, email
  providerId    String?
  role          SystemRole @default(USER)
  subscription  SubscriptionPlan @default(FREE)
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  bandMembers       BandMember[]
  orgMembers        OrganizationMember[]
  follows           Follow[]
  bookmarks         Bookmark[]
  donations         Donation[]
}

enum SystemRole {
  USER
  ADMIN
}

enum SubscriptionPlan {
  FREE
  PREMIUM
}

// ============================================
// 조직 (동아리 / 연합)
// ============================================
model Organization {
  id            String   @id @default(cuid())
  name          String
  type          OrgType
  description   String?
  coverImage    String?
  logoImage     String?
  school        String?  // 소속 학교 (해당 시)
  region        String?  // 지역
  foundedYear   Int?
  snsLinks      Json?    // { instagram: "...", youtube: "..." }
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  bands         Band[]
  members       OrganizationMember[]
  announcements Announcement[]
  events        OrgEvent[]
}

enum OrgType {
  CLUB        // 동아리
  UNION       // 연합
  CREW        // 기획팀/크루
}

model OrganizationMember {
  id             String   @id @default(cuid())
  userId         String
  organizationId String
  role           OrgRole  @default(MEMBER)
  joinedAt       DateTime @default(now())
```

## 12. 사용자 시나리오

### 12.1 일반 사용자 (팬/관객)

**시나리오 A: 이번 주말 공연 찾기**
1. 홈 피드 진입, "다가오는 공연" 섹션에서 이번 주 공연 카드들을 둘러봄
2. 관심 가는 공연 카드를 탭하여 공연 상세 페이지 진입
3. 출연 밴드 목록에서 각 밴드 프로필을 확인하고 음악 스타일 파악
4. 공연장 위치를 지도에서 확인
5. "관심 표시" 버튼을 눌러 북마크하고, "캘린더 추가"로 일정에 등록
6. 공연 포스터를 SNS에 공유

**시나리오 B: 좋아하는 밴드 팔로우 및 후원**
1. 검색에서 밴드명으로 검색하여 밴드 프로필 진입
2. 과거 공연 이력과 포트폴리오(영상/사진) 열람
3. 팔로우 버튼을 눌러 향후 소식 알림 설정
4. (프리미엄 밴드의 경우) 후원 버튼으로 소액 후원 전달

### 12.2 밴드 관리자

**시나리오 A: 밴드 등록 및 공연 기획**
1. 회원가입 후 "밴드 생성" 버튼으로 밴드 프로필 생성
2. 밴드명, 장르, 소개, 대표 사진 등록
3. 멤버 초대 (이메일/링크)로 밴드원 등록, 각 멤버 파트 지정
4. 소속 동아리(조직)에 밴드 연결 요청
5. "공연 등록"으로 새 공연 생성 - 공연명, 일시, 공연장 선택, 포스터 업로드
6. 다른 밴드를 출연진으로 초대

**시나리오 B: 공연장 예약**
1. "공연장" 탭에서 대전 지역 공연장 목록 탐색
2. 원하는 공연장의 상세 페이지에서 시설 정보와 장비 목록 확인
3. 캘린더에서 가용 날짜 확인 후 원하는 날짜 선택
4. 예약 요청 폼 작성 (공연 유형, 예상 관객 수, 요청 사항)
5. 예약 관리 페이지에서 승인 대기 상태 확인
6. 공연장 측 승인 시 알림 수신, 확정 내역 확인

**시나리오 C: 공연 후 아카이빙**
1. 공연 완료 후 공연 상세 페이지에서 "공연 기록 추가"
2. 공연 사진, 영상 업로드
3. 밴드 프로필의 공연 이력에 자동 반영
4. 피드에 공연 완료 소식 자동 게시

### 12.3 조직 관리자 (동아리 회장/운영진)

**시나리오 A: 동아리 개설 및 밴드 관리**
1. "조직 생성"으로 동아리 페이지 개설
2. 동아리명, 소속 학교, 소개, 로고 등록
3. 기존에 등록된 산하 밴드들을 조직에 연결
4. 멤버 초대 링크 생성하여 동아리원 전체 초대
5. 각 멤버에게 역할 부여 (관리자/멤버)

**시나리오 B: MT 기획 및 일정 관리**
1. 조직 페이지 > 일정 탭에서 "MT 일정 등록"
2. MT 날짜, 장소, 세부 일정 입력
3. 전체 멤버에게 일정 알림 전송
4. 멤버들이 참석 여부(참석/불참/미정) 투표
5. 참석 현황 실시간 확인
6. 공지 게시판에 MT 관련 안내사항 게시

**시나리오 C: 정기 공연 조율**
1. 조직 페이지에서 정기 공연 일정 등록
2. 산하 밴드들에게 출연 의사 확인 공지 게시
3. 참여 밴드 확정 후 공연장 예약 진행
4. 공연 등록 시 조직명 + 참여 밴드 전체를 출연진으로 설정

### 12.4 공연장 관리자

**시나리오 A: 공연장 등록 및 운영**
1. "공연장 등록"으로 공연장 프로필 생성
2. 공연장명, 주소, 수용 인원, 시설 사진 등록
3. 보유 장비 목록 등록 (앰프, 마이크, 드럼 등)
4. 대관 요금 정보 입력
5. 운영 가능 일정 캘린더에 등록 (날짜별 가용/불가)

**시나리오 B: 예약 요청 처리**
1. 예약 대시보드에서 새로운 예약 요청 알림 확인
2. 요청한 밴드의 프로필과 테크라이더 확인
3. 일정 확인 후 승인 또는 거절 (사유 메시지 첨부)
4. 승인 시 해당 날짜 캘린더에 자동 반영
5. 월간 예약 현황과 가동률 통계 확인

## 14. 경쟁사 및 유사 서비스 분석

### 14.1 직접 경쟁 서비스

| 서비스 | 설명 | 강점 | 약점 | 차별화 포인트 |
|--------|------|------|------|--------------|
| **밴드플랫폼(해외: BandMix, Vampr)** | 밴드 멤버 매칭 플랫폼 | 글로벌 네트워크, 멤버 매칭 알고리즘 | 한국 시장 부재, 조직 구조 미지원, 공연장 연결 없음 | 우리는 조직 계층 구조와 공연장 예약을 통합 제공 |
| **인디스트릿** | 국내 인디음악 커뮤니티 | 인디 음악 전문, 커뮤니티 활성화 | 밴드 관리 기능 부족, 공연장 예약 미지원 | 우리는 관리 도구 + 예약 시스템을 제공하여 단순 커뮤니티를 넘어선 운영 플랫폼 |
| **공연장닷컴** | 공연장 정보/예약 플랫폼 | 공연장 DB, 예약 시스템 | 밴드 생태계 연결 부재, 동아리 구조 미지원 | 우리는 밴드-조직-공연장을 계층적으로 연결 |

### 14.2 간접 경쟁 / 대체재

| 서비스 | 사용 방식 | 한계점 |
|--------|-----------|--------|
| **카카오톡 그룹채팅** | 밴드/동아리 소통, 공연 조율 | 히스토리 유실, 구조화 불가, 공연 기록 관리 불가 |
| **인스타그램** | 밴드 홍보, 공연 홍보 | 발견성 낮음, 팔로워 기반 제한적 노출, 관리 기능 없음 |
| **네이버 카페** | 동아리 커뮤니티 | UI/UX 낙후, 밴드 프로필 구조 부재, 모바일 경험 미흡 |
| **에브리타임** | 대학생 커뮤니티 내 동아리 홍보 | 밴드 특화 기능 없음, 외부 사용자 접근 불가 |
| **구글 캘린더 / 노션** | 일정 관리, 정보 정리 | 범용 도구라 밴드 특화 기능 없음, 외부 공개 불가 |

### 14.3 경쟁 우위 요약

1. **조직 계층 구조**: 동아리/연합 -> 밴드 -> 공연의 계층 관리를 지원하는 유일한 플랫폼
2. **올인원 운영 도구**: 프로필 + 공연 관리 + 예약 + 소통이 한 플랫폼에 통합
3. **지역 특화**: 대전 밴드 생태계에 집중하여 밀도 높은 네트워크 구축
4. **공연장 연결**: 밴드-공연장 간 직접 예약 시스템으로 양측 효율 개선
5. **아카이빙**: 공연 기록, 포트폴리오가 누적되어 밴드의 자산이 됨

## 15. KPI 및 성과 지표

### 15.1 핵심 KPI

| 구분 | 지표 | 목표 (베타 3개월) | 목표 (정식 출시 6개월) | 측정 방법 |

| **수익** | MRR (월간 반복 수익) | - | 500,000원 | 프리미엄 구독 수익 |

### 15.2 단계별 핵심 목표

**Phase 1: 베타 (0~3개월)**
- 핵심 목표: **공급 확보** (밴드, 조직, 공연장 등록)
- 집중 지표: 등록 밴드 수, 등록 조직 수, 등록 공연장 수
- 성공 기준: 대전 내 주요 동아리 3곳 이상 온보딩, 공연장 5곳 이상 등록

**Phase 2: 초기 성장 (3~6개월)**
- 핵심 목표: **활성화** (실제 예약 및 공연 등록 발생)
- 집중 지표: 월간 예약 건수, 월간 공연 등록 수, MAU
- 성공 기준: 월 예약 30건 이상, MAU 300명 이상

**Phase 3: 수익화 (6~12개월)**
- 핵심 목표: **프리미엄 전환 및 수익 발생**
- 집중 지표: 프리미엄 전환율, MRR, 예약 수수료 수익
- 성공 기준: 프리미엄 전환율 5% 이상, MRR 100만원 달성

### 15.3 모니터링 대시보드 구성 (내부용)

| 대시보드 | 포함 지표 |
|----------|-----------|
| **일일 현황** | DAU, 신규 가입, 신규 공연 등록, 신규 예약 요청 |
| **주간 리포트** | WAU, 주간 예약 전환율, 주간 피드 게시물 수, 주간 신규 밴드 수 |
| **월간 리포트** | MAU, MRR, 월간 예약 건수, 공연장 가동률, 밴드 증가 추이, 리텐션율 |
| **퍼널 분석** | 가입 -> 밴드 생성/가입 -> 공연 등록 -> 예약 요청 각 단계 전환율 |
